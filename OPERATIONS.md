# OPERATIONS.md — ручные операции над пайплайном

Инструкция описывает две ручные операции поверх автоматического пайплайна
[run_all.py](run_all.py):

1. [Ролловер фьючерса](#1-ролловер-фьючерса) — переход на новый контракт (квартальный).
2. [Смена количества контрактов с открытием нового ордера](#2-смена-количества-контрактов).

Обе операции затрагивают только торговую часть — аналитика и прогнозы продолжают
работать как есть.

---

## 1. Ролловер фьючерса

### Тайминг и почему так

- Экспирация квартального фьючерса на MOEX наступает в **19:00** дня экспирации
  — после этого старый контракт перестаёт торговаться.
- [run_all.py](run_all.py) запускается по Task Scheduler в **21:00:05**, то есть
  **уже после** экспирации.
- Значит исполнять ролловер-трейд (закрыть старый + открыть новый) надо в
  **вечернюю сессию накануне дня экспирации** — 21:00:05 дня `N−1`, когда
  старый контракт ещё жив.
- В сам день экспирации (день `N`) к моменту 21:00:05 старого контракта уже
  нет, поэтому к этому времени `ticker_close` нужно выровнять с `ticker_open`.

### Коды контрактов

Коды квартальных фьючерсов MOEX (месяц истечения): `H` = март, `M` = июнь,
`U` = сентябрь, `Z` = декабрь. Суффикс — последняя цифра года.

| Тикер | Текущий контракт | Следующий контракт |
| ----- | ---------------- | ------------------ |
| RTS   | `RIM6` (июнь 26) | `RIU6` (сент. 26)  |
| MIX   | `MXM6` (июнь 26) | `MXU6` (сент. 26)  |

### Как ролловер устроен в коде

Ролловер задаётся рассинхронизацией двух ключей в `<ticker>/settings.yaml`:

```yaml
common:
  ticker_close: 'RIM6'   # где уже есть позиция (старый контракт)
  ticker_open:  'RIU6'   # куда открываем целевую позицию (новый контракт)
```

Трейд-скрипт [trade/trade_rts_combo_SPBFUT192yc_ebs.py](trade/trade_rts_combo_SPBFUT192yc_ebs.py)
(аналогично для mix) при рассинхронизации:

1. Откроет целевую позицию в `ticker_open` (текущая там = 0 → дельта = цель).
2. Отдельным ордером закроет остаток в `ticker_close`
   (строки [trade/trade_rts_combo_SPBFUT192yc_ebs.py:241-251](trade/trade_rts_combo_SPBFUT192yc_ebs.py#L241-L251)).

### 1.1. За 1–2 дня до дня экспирации — подготовка

Правок в settings.yaml ещё **не делаем**. Но в LUA-скрипте QUIK нужно
добавить новые контракты заранее, иначе минутные свечи по ним не будут
экспортироваться и `download_minutes_to_db.py` не добьёт последние 15 минут
сессии нового контракта.

- [ ] Сверить коды новых контрактов на MOEX (quik: таблица «Текущие торги»).
- [ ] **Добавить новые тикеры в [trade/quik_export_minutes.lua](trade/quik_export_minutes.lua)**
      — список `TICKERS` на строке [trade/quik_export_minutes.lua:22](trade/quik_export_minutes.lua#L22):
      ```lua
      local TICKERS = {"RIM6", "RIU6", "MXM6", "MXU6"}  -- добавить "RIZ6", "MXZ6" при следующем ролловере
      ```
      Держим старый и новый контракт одновременно — на день `N−1` оба ещё
      торгуются. После дня экспирации старый можно убрать из списка при
      следующем редактировании (но не обязательно — нагрузки от DataSource
      почти нет).
- [ ] **Перезапустить LUA-скрипт в QUIK**: Сервисы → Lua-скрипты → выбрать
      `quik_export_minutes.lua` → Остановить → Запустить. Без рестарта
      новые тикеры не подхватятся (они регистрируются в `OnInit`).
- [ ] [trade/quik_export_positions.lua](trade/quik_export_positions.lua)
      перезапускать **не нужно** — список тикеров там не захардкожен, он
      читает всю таблицу `futures_client_holding` по счёту.
- [ ] Убедиться, что новый контракт уже торгуется и виден в экспорте позиций
      QUIK (`trade/quik_export/positions.json`).
- [ ] Если используется ручной override в
      [trade/state/positions.yaml](trade/state/positions.yaml) — убедиться, что
      он отражает реальную позицию в старом контракте.
- [ ] Проверить, что предыдущий день закрылся штатно (лог
      `trade/log/trade_<ticker>_combo_*.txt` без ошибок, done-маркер за вчера
      присутствует в `trade/state/`).

### 1.2. Накануне дня экспирации (день N−1), **до 21:00:05** — исполнить ролловер

Правим `ticker_open` на новый контракт, `ticker_close` оставляем старым.
Оба тикера (RTS + MIX) редактируем **симметрично**, чтобы оба `.tri` ушли в
QUIK одним запуском в 21:00:05.

1. В [rts/settings.yaml](rts/settings.yaml), секция `common`:
   ```yaml
   ticker_close: 'RIM6'   # остаётся старый
   ticker_open:  'RIU6'   # <-- меняем на новый
   ```
2. В [mix/settings.yaml](mix/settings.yaml), секция `common`:
   ```yaml
   ticker_close: 'MXM6'
   ticker_open:  'MXU6'   # <-- меняем на новый
   ```
3. `trade/settings.yaml` **трогать не нужно** — коды фьючерсов там не хранятся.
4. Если был ручной override — актуализировать
   [trade/state/positions.yaml](trade/state/positions.yaml) под фактическую
   позицию в старом контракте.
5. Дождаться штатного запуска в 21:00:05 этого же дня `N−1` (или запустить
   `python run_all.py` вручную — [prepare.py](prepare.py) до 21:00 зачистит
   сегодняшние прогнозы и done-маркеры, пайплайн пройдёт заново).
6. Трейд-скрипт в вечерней сессии `N−1` (21:00 и позже, старый контракт ещё
   жив до 19:00 дня `N`):
   - Откроет в новом контракте целевую позицию.
   - Закроет остаток в старом контракте.
7. После исполнения проверить в QUIK и в
   `trade/log/trade_*_combo_*.txt`:
   - В `RIM6`/`MXM6` позиция = 0.
   - В `RIU6`/`MXU6` позиция = целевая (знак по прогнозу).
   - В логе есть строка `Ролловер: закрытие позиции ... контрактов ...`.

### 1.3. В день экспирации (день N), **до 21:00:05** — выравнять `ticker_close`

К моменту запуска пайплайна в 21:00:05 старый контракт уже мёртв (экспирация
в 19:00). Чтобы трейд-скрипт не пытался проверять позицию в мёртвом тикере,
меняем `ticker_close` на новый — теперь он равен `ticker_open`.

1. В [rts/settings.yaml](rts/settings.yaml): `ticker_close: 'RIU6'`.
2. В [mix/settings.yaml](mix/settings.yaml): `ticker_close: 'MXU6'`.

После этого блок ролловера в трейд-скрипте больше не срабатывает
(`ticker_close == ticker_open`), пайплайн работает только с новым контрактом.

**LUA (опционально):** старый контракт из `TICKERS` в
[trade/quik_export_minutes.lua](trade/quik_export_minutes.lua#L22) можно
убрать в любой момент после дня `N` и перезапустить скрипт. Если оставить —
DataSource по истёкшему тикеру просто не вернёт новых баров, ошибок не
будет.

> Если забыли выровнять до 21:00:05 дня `N` — критично ничего не сломается:
> `ticker_close` указывает на уже истёкший контракт, позиция в нём = 0,
> блок закрытия старого допишет 0 ордеров. Но оставлять такое состояние
> неправильно — при следующем запуске `read_positions` может ругнуться на
> неизвестный тикер. Выровняйте при первой возможности.

### 1.4. Откат / аварийный сценарий

Если вечером `N−1` пайплайн упал до того, как трейд-скрипт успел записать
`.tri`:

- Done-маркер `trade/state/<ticker>_<trade_account>_<YYYY-MM-DD>.done`
  **не создан** — скрипт можно перезапустить повторно, он допишет заявку
  (старый контракт ещё жив до 19:00 дня `N`).
- Если done-маркер уже создан, но заявки в `input.tri` визуально нет —
  удалить маркер вручную и запустить трейд-скрипт ещё раз:
  ```bash
  rm trade/state/rts_SPBFUT192yc_2026-04-19.done
  python trade/trade_rts_combo_SPBFUT192yc_ebs.py
  ```
- Если ролловер не успели провести до 19:00 дня `N` (экспирация прошла,
  старый контракт исчез) — позиция в старом автоматически закроется по
  цене клиринга. Далее в день `N` нужно вручную открыть целевую позицию
  в новом контракте (напрямую в QUIK либо через трейд-скрипт после
  выравнивания `ticker_close = ticker_open = <новый>`).

## 2. Смена количества контрактов

Цель: изменить размер целевой позиции (например, с 2 на 3) и допослать ордер в
QUIK на дельту, **не меняя направление**.

Количество торгуемых контрактов задаётся в
[trade/settings.yaml](trade/settings.yaml) — отдельно для каждого счёта
(`accounts.iis` / `accounts.ebs`) и каждого тикера (`rts` / `mix`):

```yaml
accounts:
  ebs:
    trade_path: 'C:/QUIK_VTB_2025_ЕБС/algotrade'
    trade_account: 'SPBFUT192yc'
    rts:
      quantity_close: 2   # резерв; код читает только quantity_open, но держим в синхроне
      quantity_open:  2   # <-- основной параметр, размер целевой позиции
    mix:
      quantity_close: 1
      quantity_open:  2
```

Трейд-скрипт на [trade/trade_rts_combo_SPBFUT192yc_ebs.py:48](trade/trade_rts_combo_SPBFUT192yc_ebs.py#L48)
читает `quantity_open` и вычисляет `target_position = ±quantity_open`
(знак — из прогноза). Дальше `delta = target − current` и на дельту пишет
ордер.

### 2.1. Порядок действий (в течение дня, после штатного 21:00 запуска)

1. **Отредактировать** [trade/settings.yaml](trade/settings.yaml): поднять
   `quantity_open` в нужной секции `accounts.<key>.<ticker>`. `quantity_close`
   выровнять на то же значение (для чистоты, код его пока не использует).

2. **Удалить done-маркер сегодняшнего дня** — иначе скрипт посчитает, что
   сегодня уже отработал и выйдет:
   ```bash
   rm trade/state/<ticker_lc>_<trade_account>_<YYYY-MM-DD>.done
   # пример:
   rm trade/state/rts_SPBFUT192yc_2026-04-19.done
   ```

3. **Запустить только торговый скрипт** (не весь `run_all.py`):
   ```bash
   python trade/trade_rts_combo_SPBFUT192yc_ebs.py
   ```
   Скрипт:
   - Прочитает текущую позицию из `trade/quik_export/positions.json` (через
     [trade/read_positions.py](trade/read_positions.py)).
   - Посчитает дельту `новый_quantity_open − текущая`.
   - Допишет в `trade/<trade_path>/input.tri` **только недостающий** ордер
     (например, «Покупка 1 контракт», если было +2, стало +3).
   - Создаст done-маркер заново.

4. **Проверить** в QUIK-логе и в `trade/log/trade_<ticker>_combo_*.txt`, что
   заявка ушла и позиция совпала с целью.

### 2.2. Особые случаи

- **Прогноз на сегодня = `skip`** → целевая позиция = 0 независимо от
  `quantity_open`. Смена количества даст эффект только с ближайшего дня, когда
  прогноз снова `up`/`down`.
- **Уменьшение количества** (например, 3 → 2): скрипт запишет встречную
  заявку на дельту (продажа/покупка 1 контракт). Поведение симметрично.
- **Смена направления в тот же день** — делать через тот же механизм не нужно:
  направление определяется только прогнозом, а при смене прогноза трейд-скрипт
  сам закроет противоположную позицию и откроет новую
  ([trade/trade_rts_combo_SPBFUT192yc_ebs.py:208-238](trade/trade_rts_combo_SPBFUT192yc_ebs.py#L208-L238)).
- **Принудительный полный пересчёт до 21:00** — можно просто запустить
  `python prepare.py` вместо ручного удаления маркера: он снесёт сегодняшние
  прогнозы и все done-маркеры. Но тогда придётся гонять и весь
  `run_all.py`, а не только трейд-скрипт.

### 2.3. Чего делать НЕ нужно

- Не менять `quantity_test` в `<ticker>/settings.yaml` — он используется только
  в бэктестах и аналитике, к реальной торговле отношения не имеет.
- Не амендить (`git commit --amend`) уже запушенные ночные прогоны — для
  аудита важна цепочка done-маркеров и логов.
- Не править `input.tri` руками — QUIK перечитывает файл и может исполнить
  правку не так, как ожидаешь. Всегда идти через трейд-скрипт.
